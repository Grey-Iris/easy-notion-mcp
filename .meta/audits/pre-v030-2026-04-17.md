# Pre-v0.3.0 Health Audit

**Date:** 2026-04-17
**Branch audited:** `dev` @ `d66eb47` (9 commits ahead of `main`)
**Scope:** ~28 MCP tools in `src/server.ts`, shared infrastructure (markdown↔blocks, notion-client wrappers, file-upload, OAuth token store, HTTP transport), test suite, runtime behavior.
**Method:** independent code read, two delegated breadth scans (benchmark-bug verification, test-coverage map), runtime smoke checks against real Notion via the `.env` token. Written cold without reading parallel session work.

## TL;DR

The codebase is internally coherent and the recent PRs (#21, #22) added genuinely useful surface area with thoughtful guardrails. But the **silent-failure pattern that PR #21's runtime evidence surfaced is structural, not isolated** — at least eight other places in the existing tool surface drop, alias, or misreport user data with no signal. Several of these affect the most-used path (database entries, page content read/write). Two non-atomic mutation patterns (`replace_content`, `update_section`) can destroy a user's page if the second half of the operation fails. The test suite passes 225/225 and CI is green, but coverage is shallow: of the 28 tools, only `create_page` and `create_page_from_file` have what could honestly be called integration tests; ~21 tools have **zero** handler-level tests.

The build and runtime are healthy: `npm run typecheck`, `npm run build`, and `npm test` all pass; stdio mode boots and serves all 28 tools; HTTP mode serves the health endpoint and OAuth metadata correctly; a real-credential `get_me` + `list_databases` call round-trips successfully.

**Recommendation: do not ship as `v0.3.0` without addressing at least the four "fix now" items below. Cutting `v0.3.0-rc1` and waiting on the silent-failure audit (`audit-existing-tools-for-silentf`) before a stable tag is defensible.** Most users will never hit these edges in the demo path. The ones who do will lose data quietly.

## Findings by severity

### CRITICAL — fix before any v0.3.0 tag

**C1. Non-atomic delete-then-append destroys page content on failure** — `src/server.ts:1019-1023` (`replace_content`), `src/server.ts:1067-1076` (`update_section`).

Both tools delete every targeted block in a sequential loop, then append the new blocks. If the append phase fails — markdown parse error, Notion API rejection (e.g. >100 children, 2000-char rich-text overflow, invalid code-block language; see C3, H4, H5), rate-limit, network blip — the user's page is left **emptier than it started**. There is no transaction, no rollback, no "save to draft and retry" path. The `replace_content` description ("Deletes existing blocks and writes new ones.") tells the agent what happens but doesn't warn the agent that the delete is irreversible if the append fails. `update_section` is worse because it hides the same delete-then-append behind a more targeted-sounding name.

This is the exact same shape as the silent-failure pattern PR #21 surfaced: a destructive operation succeeds at the destruction step, then partially or fully fails at the constructive step, and the user sees only the constructive-step error. The tool description on `replace_content` doesn't warn about this; the tool description on `update_section` doesn't either.

Smallest defensible fix: capture the existing block IDs *and content* before deletion; if append throws, attempt a re-append of the originals from the captured snapshot before re-throwing. This converts "destroyed" into "best-effort restored, plus original error surfaced." Larger fix: build the new blocks first and verify they parse before touching the old ones.

**C2. Single bad `file://` upload aborts entire `create_page` / `append_content` / `replace_content` / `update_section`** — `src/file-upload.ts:78-83`.

`processFileUploads` runs all uploads via `Promise.all`. The first rejection (file not found, file too large, MIME unknown, network blip during upload) cancels the whole markdown call. From the agent's perspective, a 50-block page with one bad image URL fails to create at all — instead of creating with a placeholder or a per-image error annotation. This is on the tasuku ready queue (`single-bad-file-upload-id-fails`) but the severity is real: agents that batch uploads (a research dump, a slide-deck import) will lose entire jobs to one bad path.

Smallest fix: `Promise.allSettled`, then for each rejected upload either drop the file from the markdown (and emit a warning in the tool response) or substitute a placeholder block referencing the original `file://` URL with an error note. The `add_database_entries` handler at `src/server.ts:1339-1349` already models the right pattern — `{succeeded: [], failed: []}` — and the same shape should propagate to the page-creation response.

**C3. No depth limit on `fetchBlocksRecursive`** — `src/server.ts:315-339`.

`fetchBlocksRecursive` recursively descends into every `has_children` block with no max-depth guard. Notion does not enforce a hard cap on nesting, and `read_page` is the most-called read path. A malicious or accidentally-pathological page (deep toggle/column nesting, recursive-feeling structures) can either exhaust the call stack or, more realistically, generate hundreds of API calls and stall the agent for minutes before MCP timeout. This is on the tasuku ready queue (`depth-limit`) and the case for shipping `v0.3.0` without a guard here is weak — the fix is one parameter and one comparison.

Smallest fix: pass a `depth` argument, default cap at e.g. 6, return a "[content truncated at depth N]" placeholder block when reached. The `max_blocks` parameter on `read_page` already shows the ergonomic precedent.

### HIGH — should fix before tagging stable, can ship if explicitly documented as known limitation in `0.3.0-rc1`

**H1. `convertPropertyValues` silently drops unknown property names** — `src/notion-client.ts:201-203`.

```ts
const propConfig = ds.properties[key];
if (!propConfig) {
  continue;
}
```

If the agent calls `add_database_entry({Status: "Done", Statuz: "Bug"})`, the typo property is dropped without comment and the entry is created with only the recognized properties. The response (`{id, url}`) looks like full success. Same wrapper is used by `add_database_entries` and `update_database_entry`, so the same bug exists in three tools. This is **the textbook silent-failure mode** that the audit task was created to find — pattern-identical to the PR #21 row-reassignment finding, except this one is in the most-used database path, not a rarely-touched schema path.

Smallest fix: collect unrecognized keys; if any, throw a descriptive error naming the unrecognized key(s) and listing the valid ones from the schema. The schema is already in hand from the same `getCachedSchema` call. Cost: a few lines.

**H2. `convertPropertyValues` silently drops known-but-unhandled property TYPES** — `src/notion-client.ts:243` (`default: break`).

The switch handles 11 of Notion's ~22 property types. `relation`, `people`, `files`, `formula`, `rollup`, `created_time`, `last_edited_time`, `created_by`, `last_edited_by`, `unique_id`, and `verification` all fall through to `default: break;` — silently. An agent passing `{ Owner: "user-id" }` for a `people` property gets no error and no people-link; the entry is created without the field set. Note that `relation_property.test.ts` exists at the unit level, but the `convertPropertyValues` switch does not contain a `relation` case, so that test is exercising a path that is structurally broken in the production switch (worth checking — see Surprises §1).

Smallest fix: throw on unsupported types with a clear message ("Property 'Owner' has type 'people'; this server does not yet support setting people fields. Use the Notion UI for now."). This converts a silent feature gap into a directed error agents can recover from.

**H3. `schemaToProperties` drops unknown types AND `create_database` response lies about which properties were actually created** — `src/notion-client.ts:184` (`default: break`), `src/server.ts:1260` (`properties: schema.map(s => s.name)`).

`schemaToProperties` switches on the type string and silently skips unknown types. The handler's response then constructs the `properties` field by mapping over the *requested* schema, not the *created* schema. Net result: an agent calls `create_database({schema: [{name: "Title", type: "title"}, {name: "Owner", type: "people"}]})`, the database is created with only `Title`, and the response shows `properties: ["Title", "Owner"]` — falsely confirming that `Owner` exists. Subsequent calls to `add_database_entry` referencing `Owner` then fail with the silent drop in H1.

Smallest fix: derive the response's `properties` field from `result.properties` (the actual API response), not the input schema.

**H4. No 2000-char rich-text splitting** — `src/markdown-to-blocks.ts:20-41` (`createRichText`).

Notion's API rejects single rich-text segments over 2000 characters. The converter never splits. A single long paragraph (research note, AI-generated summary, copy-pasted article) above 2000 chars produces one segment and triggers a hard API error from `pages.create` / `blocks.children.append`. Combined with C1 in `replace_content`, this means a long markdown replacement can wipe a page and then fail to write the new content. On the tasuku ready queue (`handle-notion-char-rich-text-lim`).

Smallest fix: in `createRichText` (or just before constructing the block), split content >1990 chars into multiple text segments at word boundaries, preserving annotations.

**H5. No batching in `createPage`** — `src/notion-client.ts:251-274`.

`appendBlocks` and `appendBlocksAfter` correctly chunk children into 100-block batches. `createPage` does not — it sends `children: blocks` directly to `pages.create`. Notion rejects >100 children in one call. The asymmetry is hidden because most demo workflows are small, but `create_page_from_file` (PR #22) just made it easy to dump a 200-block markdown file in one shot. On the tasuku ready queue (`batch-block-appends-at-block-lim`).

Smallest fix: in `createPage`, if `blocks.length > 100`, do a `pages.create` with the first 100 and `appendBlocks` with the rest, then return the first response with the second response's IDs concatenated to `.results` for fidelity. Or: leave `pages.create` with up-to-100 children and document the cap.

**H6. Code-block language passed verbatim to Notion** — `src/markdown-to-blocks.ts:561` (`language: token.lang || "plain text"`).

Notion's API accepts a closed enum of languages. The marked tokenizer hands us whatever the user wrote in the fence (` ```ts `, ` ```py `, ` ```kotlin-script `). On the tasuku ready queue as `validatesanitize-code-block-lang` and `fix-markdowntoblocks-language-ali`, with the latter explicitly noting the `ts → typescript` bug was discovered during PR #22's runtime evidence and **never fixed**. So this has been a known data-loss vector for at least two weeks.

Smallest fix: a `LANGUAGE_ALIASES: Record<string, string>` map (`ts → typescript`, `js → javascript`, `py → python`, `sh → shell`, `yml → yaml`, etc.) plus a fallback to `"plain text"` for anything not in the closed enum, with a one-time `console.error` log when an unknown language is downgraded.

**H7. `read_page` silently drops unknown block types and the `column_list` / `column` round-trip is incomplete** — `src/server.ts:122-279` (`normalizeBlock`), `src/server.ts:198-207`.

`normalizeBlock` returns `null` for any block type it doesn't enumerate; `fetchBlocksRecursive` filters those out (`src/server.ts:323-326`). So a page containing a `synced_block`, `link_to_page`, `child_page`, `child_database`, `breadcrumb`, `link_preview`, `pdf`, `template`, `unsupported`, or any block type Notion adds in the future is read as if it didn't exist. The `read_page` description claims "round-trips cleanly" — it doesn't, for any of these. (CLAUDE.md line 116 makes the same claim and is also incorrect for the dropped types.)

Separately, `normalizeBlock` for `column_list` and `column` always returns `{children: []}` (lines 199-207) and relies on `attachChildren` to populate them later via the recursive walk. That is correct for `read_page`, but `duplicate_page` then passes the resulting blocks straight to `createPage` (`src/server.ts:1163`) — and Notion's `pages.create` validation for nested column_list children may not match what `attachChildren` produced. This path is **completely untested**.

Smallest fix on dropped types: emit a `paragraph` block with `[Unsupported block: <type> — content not available via this tool. View in Notion.]` so agents can at least surface that something exists. Or document the truncation honestly.

**H8. `update_data_source` response shape is unverified** — `src/server.ts:1281-1283`.

```ts
title: title ?? result.title?.[0]?.plain_text ?? "",
url: result.url,
properties: Object.keys(result.properties ?? {}),
```

`result` is what `client.dataSources.update()` returned — a data-source object, not a database object. There is no test asserting the shape of this response (`tests/update-data-source.test.ts` only asserts the request body, not the response handling). `result.url` may be `undefined`; `result.title` may be a different shape from `db.title`. Plan §12 didn't capture the response shape during runtime evidence — only the per-payload behavior. Worth one runtime probe before claiming the response is correct.

### MEDIUM — should land in 0.3.x but not blockers

**M1. Image alt text dropped on round-trip** — `src/markdown-to-blocks.ts:455-476` and `src/blocks-to-markdown.ts:206-211`. The `NotionBlock` type at `src/types.ts:84-88` has no caption field on the image variant either, so there's literally no place to store the alt text even if we wanted to. CLAUDE.md and `read_page`'s description both promise round-trip cleanliness; this is an active counter-example. Notion's image block does support `caption` (rich_text) — it's just not modeled in `types.ts` or threaded through.

**M2. `add_comment` response strips formatting** — `src/server.ts:1379`. The comment is stored with rich text (`blockTextToRichText(text)` in line 1376), but the response flattens to `plain_text`. Agents see "formatting was lost" when it actually wasn't. Cosmetic but confusing.

**M3. Cover supports `file://` upload, icon does not** — `src/server.ts:1184-1190`. The tool description says icon is emoji-only, so this is documented behavior. But it's an asymmetry that's likely to surprise agents who used `cover: "file:///path/to/img.png"` and reasonably expect `icon: "file:///path/to/icon.png"` to work the same way. The error from Notion when a `file://` URL hits the icon validator is opaque (per `audit-existing-tools-for-silentf` and the URL-as-ID audit).

**M4. Server version drift** — `src/server.ts:892` hardcodes `version: "0.2.0"` while `package.json` is `0.2.4`. MCP clients introspecting the server see the wrong version. This will worsen at v0.3.0 unless the source-of-truth is unified (read from `package.json` at build time, or from a generated constant).

**M5. `searchNotion` returns trashed data sources** — runtime observation. `list_databases` returned 5 leftover `runtime-probe-*` and `cache-probe` databases that PR #21 plan §12 claimed were trashed via `updateDataSource(..., {in_trash: true})`. Either Notion's `client.search` returns trashed items by default (likely — search has a separate `archived` filter) or the cleanup invariant in §12 silently failed. Either way, the production-facing `list_databases` tool is showing trashed databases as if they're live. Agent calls `list_databases`, picks one, queries it, gets surprising results.

Smallest fix: filter the search results by an explicit "not in trash" check (the `data_source` object has `in_trash` / `archived` — verify and filter). Larger fix: confirm whether the cleanup actually worked and patch `notion-client.ts:428-456` accordingly.

**M6. `updateDatabaseEntry` parent resolution comment is misleading and possibly wrong** — `src/notion-client.ts:574-579`:

```ts
const dbId = page.parent?.type === "database_id"
  ? page.parent.database_id
  : page.parent?.type === "data_source_id"
    ? page.parent.database_id  // data_source parent also exposes database_id
    : null;
```

The comment claims `data_source` parents expose `database_id`, but the code accesses `page.parent.database_id` regardless of the parent shape. This needs a runtime probe — if `data_source_id` parents only expose `data_source_id`, this returns `null` and throws "Page is not part of a database" for legitimate database entries created via the new `data_source` flow.

**M7. `update_page` cannot clear icon or cover** — `src/notion-client.ts:395-405` checks `if (props.icon)` / `if (props.cover)`, so passing `""` (or any falsy value) is treated as "leave unchanged." There's no way to remove an icon or cover via this tool. Not catastrophic; missing feature, not silent failure.

**M8. URL-as-ID errors remain unaddressed since 2026-04-09 audit** — `.meta/audits/url-as-id-errors-2026-04-09.md` documented that passing a Notion URL where an ID is expected produces an opaque `Invalid request URL.` error. No `normalizePageId` helper has been added since. Every tool taking `page_id` / `database_id` / `parent_page_id` / `block_id` is still vulnerable. README's `.env.example` invites users to paste an ID that they're going to copy from the URL bar.

### LOW — note for maintenance, not blocking

**L1. `TokenStore` re-encrypts the entire token list on every write** — `src/auth/token-store.ts:78-87`. Fine at small scale, O(n) per write at large scale. Not a problem until ~1000s of users.

**L2. OAuth token records pile up on refresh** — `src/auth/oauth-provider.ts:393-410` creates a *new* mcpToken record on every refresh and never deletes the old one. The old access tokens still verify until they expire. Mostly fine (they expire), but tokens without `expiresAt` (the refresh-token records at line 310-319) accumulate forever. Probably worth a sweep in `cleanup()`.

**L3. HTTP transport: per-session Notion client capture** — `src/http.ts:78-87`. The Notion client is captured in a closure for the lifetime of the MCP session. If OAuth refreshes the underlying token mid-session, the session keeps using the stale Notion token. Not catastrophic for short sessions; unbounded sessions could drift.

**L4. `processFileUploads` regex doesn't escape brackets in alt text** — `src/file-upload.ts:64`. `(?:!\[[^\]]*\]|...)` won't match `[file with [brackets]](file:///...)`. Edge case; users probably don't write that, but it's a silent fail (the file URL just isn't uploaded).

**L5. `find_replace` calls `pages.updateMarkdown` via `(notion as any)`** — `src/server.ts:1090-1100`. This is an undocumented (or at least under-typed) Notion endpoint. If the SDK ever drops it or changes the shape, the only signal will be a runtime crash. No test covers it. CLAUDE.md acknowledges it's the one tool that goes outside the markdown-to-blocks pipeline.

**L6. Schema cache is unbounded** — `src/notion-client.ts:42, 43`. `schemaCache` and `dataSourceIdCache` are `Map`s with TTL but no size cap. A long-running server that touches many databases will grow indefinitely. Tasuku has `schema-cache-limit` for this.

## Testing assessment

`npm test`: 225/225 passing in 11.5s. `npm run typecheck`: clean. `npm run build`: clean. CI is green per `dev`'s recent push.

Coverage is **shallow but not absent**. Of the 28 tools in `src/server.ts`:

| Tier | Count | Tools |
|---|---|---|
| Direct integration tests for the *handler* | 2 | `create_page` (parent resolution only), `create_page_from_file` |
| Mocked-handler tests (assert what the SDK was called with, not what it returns) | 3 | `create_database`, `update_data_source`, `list_databases` |
| Unit tests for the underlying helper *only*, not the handler | 1 | `update_section` (boundary detection) |
| **Zero coverage at any layer** | **22** | `append_content`, `replace_content`, `find_replace`, `read_page`, `duplicate_page`, `update_page`, `archive_page`, `search`, `list_pages`, `share_page`, `get_database`, `query_database`, `add_database_entry`, `add_database_entries`, `update_database_entry`, `list_comments`, `add_comment`, `move_page`, `restore_page`, `delete_database_entry`, `list_users`, `get_me` |

The pure conversion tests (`markdown-to-blocks.test.ts`, `blocks-to-markdown.test.ts`, `roundtrip.test.ts`) are genuinely valuable and well-constructed — that path is the project's structural backbone and the tests reflect that. Similarly `tests/file-upload.test.ts` covers the code-range exclusion logic that's easy to get wrong.

What's missing — and is the structural risk for v0.3.0:

- **Zero round-trip tests at the *tool* level.** The `roundtrip.test.ts` tests run the converters in process; there is no test that creates a page via `create_page`, reads it via `read_page`, and confirms the markdown matches. The whole project's value proposition is tool-level round-trip cleanliness, and that's the one thing untested.
- **Zero error-path tests for destructive operations.** `replace_content`, `update_section`, `archive_page`, `delete_database_entry`, `move_page` — none have tests for "what happens when the destructive step succeeds and the constructive step fails." This is the C1 finding; it's not tested because there's no infrastructure for it.
- **Mocked tests assert on call args, not outcomes.** `tests/update-data-source.test.ts` and `tests/parent-resolution.test.ts` mock the SDK and verify the right args were passed. That's useful for regression-protecting the wrapper layer, but it can't catch a Notion API change or a response-shape mismatch.
- **`relation-property.test.ts` exists but `convertPropertyValues` doesn't handle relations.** Worth checking — if the test is mocking around the gap, it's misleading.
- **`stdio-startup.test.ts` checks the banner message, not that the server actually serves tools.** A 3.9-second test with low signal.

The proportionality is wrong: the most-tested files (`markdown-to-blocks.test.ts` at 897 lines, `blocks-to-markdown.test.ts` at 570 lines) are the *least* likely to silently misbehave. The most under-tested code (the 22 untested tool handlers, the silent-drop patterns in `convertPropertyValues`) is the most likely to leak data.

`audit-existing-tools-for-silentf` and `build-ee-testing-suite-for-live` from the tasuku ready queue are exactly the right next moves.

## Positive patterns worth preserving

- **`add_database_entries` partial-success response shape** (`src/server.ts:1339-1349`): `{succeeded: [], failed: []}` with per-index error messages. This is the correct pattern for any tool that does N operations under one call. Should be the model for the C2 fix.
- **`create_page_from_file` boundary checks** (`src/read-markdown-file.ts`): absolute-path requirement, `realpath`-based workspace containment (defeats symlink escape), explicit UTF-8 fatal decode (catches the `fs.readFile` silent-replacement bug), 1 MB cap, post-`realpath` extension check. The order of operations is correct — symlink-resolve before extension-check, not after. This is the strongest input-validation surface in the project.
- **`update_data_source` schema cache invalidation** (`src/notion-client.ts:500`): the cache is invalidated on success only. Codex caught this in plan review and the implementation honors the distinction. Tested.
- **Transport-conditional tool registration** (`src/server.ts:927-945`): both `ListTools` and `CallTool` honor `transports`. The `transports` field is projected out of the wire response. The `create_page_from_file` tool is correctly hidden from HTTP clients. Good architectural fit.
- **The CLAUDE.md "Custom markdown conventions" table** is the single best onboarding artifact in the repo. It's also a contract — every block type listed there *must* round-trip, and the round-trip tests honor that.
- **The `update_data_source` tool description** is the correct response to the silent-row-reassignment finding from PR #21. It's specific, opens with `CRITICAL`, and tells the agent the order in which to safely operate. This is the model for descriptions on the C1 / C2 / C3 / H-series tools that need the same treatment.
- **`processFileUploads` correctly excludes code blocks** from file-URL substitution (`src/file-upload.ts:73-75`), so a markdown code sample showing `[file](file:///example.png)` doesn't trigger an upload. Easy to get wrong; this got it right.
- **OAuth flow is correct on the security-relevant points**: PKCE is honored (`oauth-provider.ts:264`), authorization codes are one-time use (line 287), the MCP-client–to–Notion-token mapping is encrypted at rest (AES-256-GCM in `token-store.ts`), refresh-token rotation works (line 393-410). For an OAuth implementation written in 470 lines, this is impressively focused.

## Surprises (top two)

1. **`relation-property.test.ts` exists, but `src/notion-client.ts`'s `convertPropertyValues` switch has no `relation` case.** The test (`tests/relation-property.test.ts`, 83 lines) appears to be exercising a code path that the production switch silently drops. Either the test is testing a different surface (e.g., the read path, not the write path), or the test is asserting against a mock that hides the structural gap. Worth a 5-minute read on what that test actually covers vs. what the user thinks it covers.

2. **Five test databases from PR #21's runtime evidence are still visible in `list_databases`** (`runtime-probe-payload1`, `runtime-probe-payload3-inline-{omitted,false,true}`, `cache-probe`). Plan §12 explicitly stated "All 5 test databases ... were successfully trashed." Either the trash operation didn't actually trash them, or `searchNotion` (and therefore `list_databases`) returns trashed items by default. Either way, the production tool is showing trashed databases, and PR #21's "cleanup invariant: HELD" claim deserves a re-read. A maintainer running the audit's own runtime evidence would have seen this.

## Top 3 "fix now" — one sentence each

1. **C1 — `replace_content` and `update_section` are non-atomic and can destroy page content if the append phase fails.** Capture pre-delete state, attempt restore on append failure, surface both errors.
2. **H1/H2/H3 — `add_database_entry`, `add_database_entries`, `update_database_entry`, and `create_database` silently drop unknown property names and unsupported types, and the `create_database` response falsely confirms what was dropped.** Throw on unknowns; derive response from the actual result, not the request.
3. **C3 — `read_page` has no recursion depth limit and `fetchBlocksRecursive` can be weaponized.** Add a `maxDepth` parameter, default 6, with a truncation marker block.
