## Top-line verdict

**Dev is in reasonable shape for a v0.3.0 cut, but not without changes.** No finding in this audit argues that `dev` is structurally broken. Several load-bearing claims in `CLAUDE.md` and tool descriptions are currently overstated, and the tool surface has two categories of risk that deserve a fix or an explicit user-facing warning before v0.3.0: (1) the destructive edit tools (`replace_content`, `update_section`) are structurally non-atomic on Notion's API and can half-destroy pages on mid-call failure; (2) HTTP-mode tools still honor `file://` uploads, which promotes host-filesystem confidentiality into the HTTP trust boundary — critical when compounded with static-token mode's lack of `/mcp` auth. ### Tier 1 — critical (ship-blocker unless addressed)

#### F-1 — HTTP mode honors `file://` paths → host-filesystem read primitive (compound with static-token no-auth)

**Severity:** critical in static-token-HTTP compound case; **high** in OAuth-HTTP case.
**Origin:** Codex Pass 5 BV-1 + BV-2; debate round reframed threat model.
**Path:** `src/file-upload.ts:63-80` → `src/notion-client.ts:79-100` (no root check on `file://`); reachable from `src/server.ts:964,1013,1023,1074,1185-1187` (`create_page`, `append_content`, `replace_content`, `update_section`, `update_page.cover`). Static-token `/mcp` mount without auth at `src/http.ts:189-195`; OAuth `/mcp` correctly protected at `src/http.ts:173-185`.
**Attack:**
- **Static-token mode:** any network-reachable caller `POST /mcp` → `tools/call create_page { markdown: "[loot](file:///etc/passwd)" }` → server reads file, uploads to the operator's Notion workspace (operator's token is static, so the operator's workspace receives the file). Unauthenticated arbitrary remote file read.
- **OAuth mode:** authenticated caller does the same, but the file lands in the caller's own workspace. Still a trust-boundary escalation from "can access your own Notion workspace" to "can read host files the server process can read" (SSH keys, `.env`, adjacent repos, secrets).
**Fix sketch:** gate `file://` through the same `transport === "stdio"` check used for `create_page_from_file` (`src/server.ts:491`, `940-945`). In HTTP mode, either strip `file://` URLs from markdown before upload-processing, or reject the request with a clear error. `update_page.cover` needs the same gate.

**Debate record (`audit-b-pass5-boundaries`, initial + rebuttal):**
- **Challenge:** "In OAuth mode the caller is already authenticated to their own Notion workspace — uploading `/etc/passwd` there is self-harm, not exfiltration. Who benefits from the bug?"
- **Response:** "The violated property is host-filesystem confidentiality across the HTTP trust boundary, not workspace confidentiality. The attacker could not read `/etc/passwd` before; after this, they can, because the server reads it. Delivery into the attacker's own Notion does not dissolve the escalation — it's still a remote arbitrary local-file-read primitive."
- **Resolution:** survived with severity split. OAuth-mode alone: high. Static-token mode compound (BV-1+BV-2): critical. The reframing made the threat model sharper than the original write-up.

#### F-2 — Three test files test copied logic, not production code; `tests/relation-property.test.ts` can stay green while runtime relation support is broken

**Severity:** critical on the relation-property case; high on the other two.
**Origin:** Codex Pass 6 section 4; Pass 2 SF-5 cross-corroborates.
**Affected tests:** `tests/relation-property.test.ts:3-83`, `tests/list-databases.test.ts:3-58`, `tests/update-section.test.ts:3-23`. Each defines a local copy of a helper lambda lifted from `src/server.ts` / `src/notion-client.ts` and tests that copy.

Do not go to v0.3.0 without one of these; the suite's perceived coverage is misleading.

### Tier 2 — high (fix before v0.3.0 or add prominent warning)

#### F-3 — `replace_content` and `update_section` are non-atomic delete-then-insert with no rollback (design limitation)

**Severity:** high (was critical before debate).
**Origin:** Codex Pass 3 AT-1 + AT-2; debate downgraded both.
**Path:** `src/server.ts:1016-1027` (`replace_content`): `listChildren` → `deleteBlock` per block → `processFileUploads` → `appendBlocks` in 100-block chunks. `src/server.ts:1029-1080` (`update_section`): same pattern, but also deletes the section's heading block before reinserting. Supporting wrappers `src/notion-client.ts:315-374`.
**Failure mode:** If deletion partially succeeds and append fails, the page is blank or half-deleted. If append partially succeeds, the page has mixed old+new state. A retry of `update_section` after heading deletion fails with "heading not found" because the anchor is gone. Callers get a generic error with no cursor telling them how far the operation got.
**Root cause:** Notion's API has no "replace these blocks" transactional primitive. There is no client-side rollback that gives true atomicity.
**Fix sketch — three options, stacked:**
1. **Minimal, pre-v0.3.0:** Prominent warning in the `replace_content` and `update_section` tool descriptions in `src/server.ts` AND in the README: "This tool performs a destructive edit with no rollback. If the operation fails mid-call, the page may be partially destroyed. For content you cannot afford to lose, prefer `append_content` or duplicate the page first."
2. **AT-2 specific, implementable now:** Stop deleting the heading block. Delete only `sectionBlocks.slice(1)`. If the first replacement block is also a heading, strip it. Append the rest after the original heading's id using the existing `appendBlocksAfter` wrapper. Preserves the retry anchor.
3. **Longer-term:** Best-effort snapshot. Before the destructive phase, serialize the current top-level blocks in memory and optionally return them in the error response so callers can recover. Not true rollback, but reduces blast radius.

**Debate record (`audit-b-pass3-atomicity`, initial + rebuttal):**
- **Challenge:** "Notion has no transactions; delete-then-insert is the only algorithm available. Calling this 'critical' is calling the API design critical. Defend with a concrete rollback algorithm, downgrade, or recommend tool removal."
- **Response:** "I don't defend critical. There is no true rollback on Notion's surface. A client-side snapshot is implementable but is only best-effort recovery, not atomicity. Reframe as a design limitation requiring prominent warnings. AT-2 alternative semantic (keep heading, delete slice(1), append after heading-id) IS implementable with existing wrappers."
- **Resolution:** downgraded to high; AT-2 gains a concrete fix. The debate converted a binary "is this a bug" into three actionable recommendations.

#### F-4 — HTTP static-token mode mounts `/mcp` with no authentication

**Severity:** high on its own; combined with F-1 it forms the critical case.
**Origin:** Codex Pass 5 BV-2.
**Path:** `src/http.ts:189-195` — when `useOAuth` is false (only `NOTION_TOKEN` is set, no OAuth creds), `/mcp` is mounted with zero middleware. OAuth-mode path at `src/http.ts:173-185` correctly uses `requireBearerAuth`.
**Threat:** Any network-reachable client gets full Notion API access via the operator's static token. On default `PORT=3333` and a misconfigured firewall, this means anyone on the LAN or internet gets write access to the operator's workspace.
**Fix sketch:** Either require an inbound auth header in static-token mode (simplest: a second env var `NOTION_MCP_BEARER=...` compared against `Authorization: Bearer <...>`), or refuse to bind on non-loopback interfaces unless the operator sets an explicit `NOTION_MCP_ALLOW_REMOTE=1` override. Document either choice in the README. The tests at `tests/http-transport.test.ts:45-146` currently *assert* that static-token mode accepts unauthenticated `/mcp` — those tests would need to be updated to reflect the new contract.

#### F-5 — `read_page` / `duplicate_page` silently drop unsupported Notion block types

**Severity:** medium (was critical pre-debate; Codex conceded).
**Origin:** Codex Pass 2 SF-4; debate downgraded.
**Path:** `src/server.ts:276-277` (`normalizeBlock` returns `null` for unknown types) → `src/server.ts:323-325` (filtered from results) → feeds into `read_page` at `src/server.ts:1123-1130` and `duplicate_page` at `src/server.ts:1160-1163`. `blocksToMarkdown` has a matching `default: return "";` at `src/blocks-to-markdown.ts:225-226`.
**Why this matters:** agents typically call `read_page` → modify the markdown → call `replace_content` with the modified markdown. If `read_page` silently omitted blocks (synced_block, child_database, link_to_page, newer types), `replace_content` overwrites the page using markdown that's missing content that existed before. That is a data-corruption compound, not just a read-side nit.
**Fix sketch (from debate):** Keep filtering unknown blocks on the raw side (Notion ships new types often; crashing on unknown is worse). At `normalizeBlock`'s `null` return, record `{ id, type }`. Expose that list in the `read_page` response as `warnings: [{ code: "omitted_block_types", blocks: [...] }]` and consider prepending a `<!-- omitted Notion blocks: synced_block, child_database -->` sentinel comment to the markdown so agents feeding it to `replace_content` can see the lossy read.

**Debate record (`audit-b-pass2-silent-failures`, initial + rebuttal):**
- **Challenge:** "Notion ships new block types multiple times per year. If the server didn't filter, it would crash on every Notion update. You're marking correct defensive behavior as critical. Defend, downgrade, or withdraw."
- **Response:** "Downgrade. Critical is not defensible. The fix is warn-and-continue: filter silently becomes filter-with-metadata-exposed. Concrete implementation: record omitted `{id,type}` pairs at the normalize-to-null site, return them in `read_page`/`duplicate_page` under a `warnings` field."
- **Resolution:** downgraded to medium, reframed. The debate turned a binary "this is critical" into an implementable remedy.

#### F-6 — Nested task items silently dropped in both directions

**Severity:** high.
**Origin:** Codex Pass 2 SF-1; Pass 4 RT-7 corroborates.
**Path:** Forward: `src/markdown-to-blocks.ts:149-162` — computes `children` from nested list tokens, but the `item.task` branch at 154-162 pushes a `to_do` block **without** attaching them and `continue`s. Reverse: `src/server.ts:281-312` has no `case "to_do"` in `attachChildren`, and `src/blocks-to-markdown.ts:196-199` renders `to_do` without recursion.
**Concrete failure:** User writes `- [ ] Parent\n  - child`. Server creates a flat `to_do` block with no child. Page displays the parent only. No error. The `to_do` block type in Notion does support children; the code just doesn't forward them.
**Fix sketch:** Add `children` to the `to_do` branch of `listTokenToBlocks`, add a `case "to_do"` to `attachChildren`, add recursion in `renderBlock`'s to_do case. Parallels the existing handling for bulleted/numbered list items. Small, contained change.

#### F-7 — Checkbox `Boolean("false")` → true (database entry property coercion)

**Severity:** high.
**Origin:** Codex Pass 2 SF-7.
**Path:** `src/notion-client.ts:228-229` — literally `result[key] = { checkbox: Boolean(value) };`. Any non-empty string is truthy, so `"false"`, `"0"`, `"no"` all mark the checkbox as checked.
**Attack/accident surface:** agents and scripts that round-trip through JSON stringification, or that forward user input directly, will hit this. The tool description advertises "simple key-value pairs" and gives users no hint that strings are unsafe.
**Fix sketch:** Reject non-boolean input for checkbox properties with a clear error naming the property. If string tolerance is desired, parse explicitly: accept `"true"`/`"false"`/`true`/`false` only; reject everything else.

#### F-8 — Relation and other non-whitelisted database property types silently omitted on write, nulled on read

**Severity:** high.
**Origin:** Codex Pass 2 SF-5; amplified by F-2's dead-copy test observation.
**Path:** Read: `src/server.ts:48-80` — `simplifyProperty` handles only title/rich_text/number/select/multi_select/date/checkbox/url/email/phone_number/status; everything else (relation, formula, rollup, files, created_time, last_edited_by, rollup, unique_id, etc.) hits the `default: return null;` at 79-80. Write: `src/notion-client.ts:199-245` — `convertPropertyValues` has the same whitelist and silently drops unknown types via `default: break;` at 243-244.
**Impact:** `query_database` returns rows that look complete but have `null` in every relation/formula column. `add_database_entry` / `update_database_entry` silently ignore keys users provided that target relation columns. Users debugging "why didn't my relation save?" get no diagnostic.
**Fix sketch:** add real handling for `relation` (at minimum — it's the most common user-facing gap) and either add handling for `formula`/`rollup` (read-only in Notion anyway) or explicitly document them as read-only with a null return. For `files`, `created_time`, `last_edited_by`, etc., document them as read-only. Then: reject unknown property keys on write with an error, not a silent skip.

#### F-9 — Database property typos + 5-minute schema cache staleness silently drop keys

**Severity:** high.
**Origin:** Codex Pass 2 SF-6.
**Path:** `src/notion-client.ts:42-45` (5-min TTL cache), `68-76` (returns cached until expiry), `199-203` (`if (!propConfig) continue;` silently skips unknown keys), `554-565` / `568-590` (submit only the filtered converted object).
**Impact:** User just added a property to their database. For up to 5 minutes, the cache still says the property doesn't exist. `add_database_entry` silently drops the new key. User retries, maybe the cache has expired now, maybe not. No error, no diagnostic.
**Fix sketch:** collect dropped keys in `convertPropertyValues` and either (a) return them in the tool response under `warnings.dropped_property_keys`, or (b) invalidate the cache once and retry if any input key is missing from the cached schema (self-correcting cache on miss). Option (b) is more ergonomic; option (a) is the minimum fix.

#### F-12 — Token store is not concurrency-safe; file corruption → silent `[]`

**Severity:** medium (operational sharp edge for multi-tenant HTTP).
**Origin:** Codex Pass 5 BV-5 + BV-6; Pass 3 AT-6 echoes it.
**Path:** `src/auth/token-store.ts:72-88` — load-modify-save cycle with no lock, no atomic write (`writeFile` directly, not `writeFile(tmp) + rename(tmp, final)`). `src/auth/token-store.ts:62-69` — every read/decrypt/parse error is swallowed and returns `[]`. Back-to-back writes at `src/auth/oauth-provider.ts:298-319` and `393-402` (access+refresh record pair).
**Impact:** two simultaneous `/token` exchanges race; one clobbers the other. Partial write or corruption → next read sees empty store, next write overwrites → all tokens silently invalidated.
**Fix sketch:** serialize writes behind a process-local mutex (no external dep needed), write to `tokens.json.tmp` and `fsync` + `rename` atomically. On corrupt read, fail closed (return an error, preserve the bad file as `tokens.json.corrupt.<ts>`) instead of returning `[]`.

#### F-13 — `create_page_from_file` TOCTOU between validation and read

**Severity:** medium (stdio-only, requires local attacker with FS write).
**Origin:** Codex Pass 5 BV-3.
