# OSS Dashboard Gap-Finding Pass

**Date:** 2026-04-21
**Scope:** Blind spots, wrong assumptions, and failure modes in the current dashboard plan.

---

## A. Blind Spots by Persona

**A potential contributor evaluating whether to send a PR.** They don't care about your traffic numbers or npm downloads. They want to know: how long will my PR sit before someone looks at it? Is the CI going to take 2 minutes or 20? Are there good-first-issue tasks I can actually pick up, or is every open issue implicitly assigned to the maintainer? The plan has PR/issue analytics (exploration doc §1.8, lines 86-94) but those are maintainer-facing throughput stats, not contributor-facing friction signals. A contributor wants a "contributor experience" surface: median time-to-first-review, CI duration, setup complexity (number of env vars, OS constraints), and whether the project is actually accepting outside contributions or just technically open-source. None of the 13 categories or 10 framings address contributor onboarding friction.

**A potential user (OSS dev choosing between MCP servers) visiting the dashboard.** The plan shows coverage matrices and token benchmarks, which answer "what can it do?" but not "does it actually work reliably?" A user choosing between MCP servers wants operational evidence: error rates in real usage, which MCP clients have been tested (Claude Desktop, Cursor, Cline, Continue, custom), compatibility with model versions, and whether the server handles Notion API rate limits gracefully or just crashes. The benchmark pipeline (exploration doc §1.3, lines 32-37) measures tokens, not reliability. There's no surface for runtime behavior data. The user would also distrust a dashboard with no methodology links or raw data access; the plan mentions a methodology page for benchmarks but not for any other metric.

**A security researcher who stumbled across easy-notion-mcp.** The security posture panel (exploration doc §1.10, lines 103-111) covers dependency vulnerabilities but ignores the server's own attack surface. A security researcher wants: what Notion API scopes does the token require, what's the permission model for each tool (which ones can delete data?), is there input validation on user-supplied markdown (injection via markdown-to-blocks?), and what does the OAuth token storage look like (CLAUDE.md mentions AES-256-GCM encrypted file-based storage at line 134, but the dashboard doesn't surface the encryption/rotation state). They'd also want to see if the server sanitizes content from Notion before returning it to the MCP client, since Notion page content is user-controlled and could contain prompt injection attempts. The plan has zero coverage of the server's own security posture as distinct from its dependency security.

**A future James six months from now, opening the dashboard after a month away.** His first question is not "what is the current state of everything" but "what happened while I was gone that I need to deal with?" The orchestrator's retrospective-mode framing (#3) partially addresses this, but the plan doesn't surface *events* distinct from *state*. Specifically: were any issues filed by real users (not bots)? Did any dependency go EOL or get a critical CVE? Did Notion ship a new API version? Did npm downloads spike or crater? Did someone fork the repo and build something? These are event-detection questions, not state-snapshot questions. The dashboard as planned shows current state with optional diffs, but doesn't have an event log or "things that happened" timeline that crosses all data sources into a single chronological view.

**A competing MCP maintainer (Awkoy, official Notion MCP).** The coverage matrix with competitor columns (exploration doc §1.1, line 16) is the obvious risk: it shows exactly what we haven't implemented, which they could use as their own roadmap. But the subtler risk is the tasuku board and handoff queue. If the dashboard is ever public-facing, our PM surface would show exactly what we're planning to build next, how many tasks are blocked, and how much maintenance debt we're carrying. A competitor could monitor our priorities and ship features we've queued but haven't started. The plan mentions "public-facing polish" as "probably never" (exploration doc line 363) but the orchestrator framing #4 (agent-first, MCP-exposable) means the data could leak through tool access even without a public URL.

## B. Categories the Current Plan Doesn't Touch

**Runtime telemetry.** No panel captures what happens when someone actually uses the MCP server: error rates, Notion API latency, rate limit hits, tool invocation frequency, which tools are popular vs. unused. This is the single largest category gap. The plan measures the project's health (code, deps, CI) and the project's reach (traffic, downloads) but not the project's actual behavior in production. Even basic opt-in anonymous usage telemetry (which tools get called, what error codes come back) would transform the "next to implement" ranker from guesswork to data.

**API contract stability.** The MCP tool schemas are a public contract; downstream consumers (MCP clients, wrappers) depend on them. The plan has no surface for tracking whether tool schemas changed between releases, whether any change was breaking (renamed parameter, changed type, removed tool), or whether the server's behavior for a given tool+input is consistent across versions. This matters because CLAUDE.md (line 11) explicitly calls out downstream consumer impact.

**Notion API quota/rate-limit headroom.** The Notion API has rate limits (3 requests/second per integration). The plan doesn't track how close real usage gets to those limits, whether batch operations (like reading a large database) are approaching thresholds, or what the retry/backoff behavior looks like in practice.

**Maintainer sustainability signals.** Time invested per week, ratio of reactive work (bug fixes, dep updates, security patches) to proactive work (features, docs), how many "maintenance Saturdays" in a row without shipping user-visible value. The tasuku panels show task throughput but not the human cost of that throughput. This matters because the single biggest risk to a solo-maintainer project is maintainer burnout, and the dashboard should be the tool that makes that visible before it happens.

**Downstream consumer health.** How many packages depend on easy-notion-mcp (npm dependents), are any of them broken by a recent release, are there open issues that are actually "your update broke my integration" reports? The plan mentions npm dependent count (exploration doc line 118) as a community signal, but treating it as a vanity metric misses the point: dependents are a responsibility, not a trophy.

## C. Assumptions That Might Be Wrong

**1. "A dashboard is the right form factor."** The plan assumes you'll open a browser tab and look at panels. But the orchestrator framings already hint that this might be wrong: framing #2 (narrative briefing), #4 (agent-first), and #5 (dispatch hooks) all point away from a visual dashboard toward something more like a reporting agent. What if the highest-value version of this isn't a web app at all but a CLI command (`oss-report`) that prints a 10-line summary to your terminal, or an agent that posts a weekly digest to a file, or a set of MCP tools that other agents can query? The React/Vite/Recharts stack (exploration doc §4, lines 331-337) presupposes the answer is "interactive web UI," but the actual need might be "synthesized situational awareness," which doesn't require a browser.

**2. "Local-hosted is what James wants."** The plan defaults to local hosting (exploration doc line 4: "local-hosted dev dashboard") without examining why. If the reason is cost, a static site on GitHub Pages or Cloudflare Pages costs nothing. If the reason is private data (tasuku tasks, handoffs), the question is whether the dashboard should show private data at all or whether the private data should live in a separate non-public surface. If the reason is latency to local filesystem data, a collection job that runs locally and pushes results to a static host gets the same outcome. The "local-hosted" constraint shapes every architecture decision (no CDN, no mobile access without Tailscale, no sharing with collaborators) and it was never explicitly chosen; it was inherited from github-analytics.

**3. "The 6-panel MVP is the right size."** The exploration doc proposes 6 panels for a one-day build (lines 305-319). But the panels vary wildly in value. The coverage matrix (panel 1) is a strategic planning tool you'd consult weekly. The handoff queue (panel 4) is a 30-second `ls .meta/handoffs/` you could do in a terminal. If the MVP has 6 panels and only 1 or 2 get used regularly, the unused panels create maintenance debt from day one. A stronger MVP test: what is the ONE panel that, if it existed, would change a decision you make this week? Build that one, use it for two weeks, then decide what's next. Alternatively: the right MVP might be 10 panels if they're all trivially cheap (static renders of existing data with no interactivity), because the scaffolding cost dominates and incremental panels are nearly free once the frame exists.

**4. "Multi-project data model from day one."** The exploration doc's open decision #8 (line 388) suggests designing for multi-repo from the start. But easy-notion-mcp is the only Grey-Iris project with significant complexity. Building a multi-project abstraction before the second project exists means designing the abstraction from one data point, which almost guarantees it'll be wrong. The github-analytics multi-repo support works because all repos have the same GitHub API shape. Tasuku tasks, Notion coverage matrices, and MCP tool schemas are project-specific. "Multi-project" for these panels isn't a data model choice; it's a completely different feature set.

**5. "JSON-on-disk, no database."** The plan explicitly avoids databases (exploration doc line 337). This works for point-in-time snapshots but breaks for any temporal query: "when did the CI start flaking?", "how has download count trended?", "what was the coverage matrix a month ago?" Historical data requires either a database, a growing pile of timestamped JSON files (which is a bad database), or giving up on history entirely. The plan implicitly assumes history matters (benchmark trends, traffic trends, npm download trends all appear in the feature list) while explicitly choosing a storage model that makes history painful.

**6. "Vite + React is justified by the UI complexity."** The exploration doc chooses React because github-analytics uses it (line 331). But github-analytics is a relatively simple chart-rendering app. If the dashboard is mostly static data displays with occasional filtering, a zero-JS static site generator (Astro, Eleventy) with islands of interactivity would be lighter, faster to build, and trivially deployable to any static host. React's value is in complex interactive state management; the coverage matrix filter might need it, but most panels are read-only data displays. The portability argument ("copy github-analytics components") saves maybe a day of work while committing to a heavier stack for the project's lifetime.

**7. "Displaying data is the job."** This is the deepest assumption and the one most likely to be wrong. The plan is a data-display tool: collect data, render panels, let the human interpret. But the orchestrator framings #1 (decision-support), #2 (narrative), and #5 (dispatch hooks) all point toward a tool that *acts* on data. The difference: a display tool shows "CI flake rate is 12%." A decision tool says "CI flake rate jumped from 3% to 12% this week; the three flaky tests are X, Y, Z; estimated fix effort is 2 hours; this is now your highest-leverage task." The plan never commits to which of these it wants to be, and the architecture (React SPA reading JSON) is optimized for the first while the stated ambition (orchestrator framings) points to the second.

## D. Prior Art Worth Borrowing

**Grafana's alerting model.** Grafana's core insight is that dashboards people have to remember to open don't get opened. The alert rules engine means the dashboard comes to you when something crosses a threshold. For this project: if the Notion API ships a new version, or npm audit finds a new CVE, or a GitHub issue sits unanswered for 3 days, the dashboard should notify (terminal bell, file drop, Discord webhook via agent-listener) rather than waiting for James to check. The current plan has no push mechanism at all.

**Shields.io / GitHub badge approach.** For the "potential user evaluating MCP servers" persona, the highest-leverage surface isn't a dashboard page but a set of dynamic badges in the README: coverage %, latest version, downloads/week, CI status, Notion API version supported. These are zero-click, zero-navigation signals that live where users already are (the GitHub repo page). The plan mentions shields.io only in "What I Didn't Explore" (line 424) and dismisses it as supplementary, but for external trust signals, badges in the README are 10x more visible than a separate dashboard.

**Sourcegraph's notebook/search-first model.** Instead of pre-defined panels, Sourcegraph lets you write queries and compose them into notebooks. For a solo maintainer, the most valuable dashboard feature might not be panels but a query interface: "show me everything that changed in the last week that I should care about," composed from whatever data sources are connected. This is closer to the agent-first framing (#4) than to a traditional panel dashboard.

**Sentry's issue-grouping and ownership.** Sentry doesn't just show errors; it groups them, assigns owners, tracks resolution status, and connects errors to releases. The relevant idea: every dashboard signal should be either "acknowledged and tracked" or "new and needing attention." The current plan shows data but has no concept of signal triage or acknowledgment. Once you've seen that CI flake rate is 12%, the dashboard should stop showing it as news until it changes.

**Vercel Analytics' "what matters" curation.** Vercel Analytics shows Web Vitals, not raw metrics, because they did the work of deciding which metrics actually predict user experience. For this dashboard: instead of exposing every possible data point, decide which 5 to 8 numbers actually predict "this project is healthy" vs. "something needs attention," and make those the entire surface. Everything else is drill-down. The current plan is additive (13 categories, each with 5 to 10 sub-features) when the harder and more valuable work is subtractive.

## E. Pre-Mortem: How This Dashboard Becomes Useless

**1. Death by manual data entry.** The coverage matrix (exploration doc's highest-value panel) requires hand-curated YAML (line 371). Every time a feature ships, someone has to update the YAML. In practice, the YAML falls behind within two releases, the coverage numbers become untrustworthy, and the panel gets ignored. **Design choice that invites this:** choosing hand-curated YAML over an automated coverage-detection script as the MVP data source (exploration doc open decision #1, lines 371-372). The exploration doc acknowledges this risk but defers automated refresh to v2, which means the MVP ships with the failure mode baked in.

**2. No trigger to open it.** The dashboard requires a conscious decision to open a browser tab. There is no notification, no CLI prompt, no scheduled digest, no integration with any tool James already uses daily (terminal, Claude Code, GitHub). After the novelty wears off (2 to 3 weeks), the dashboard joins the pile of local tools that technically work but never get opened. **Design choice that invites this:** building a web app with no push/notification layer (exploration doc §4 has no mention of alerts, webhooks, or CLI integration). The refresh model discussion (line 387) is about how the *page* updates, not about how the *human* gets prompted to look.

**3. Maintenance cost exceeds value.** The dashboard becomes its own project: collector scripts break when APIs change, React deps need updating, the build chain needs maintenance. For a solo maintainer already stretched across the MCP server, the dashboard becomes the thing you maintain instead of the thing that helps you maintain other things. **Design choice that invites this:** choosing a full React SPA with Express/Hono backend (exploration doc lines 331-337) when a simpler architecture (static site generator, or even a markdown report generated by a script) would have a fraction of the maintenance surface.

**4. Too many panels, no hierarchy.** The plan describes 6 MVP panels and 30+ eventual panels across 13 categories. Without aggressive curation, every panel competes equally for attention, and the dashboard becomes a wall of data where nothing stands out. The weekly briefing (orchestrator framing #2) was supposed to solve this, but if it's just another panel alongside 30 others, it doesn't. **Design choice that invites this:** the additive feature-inventory approach of the exploration doc (§1, lines 10-133), which lists everything that *could* be shown without deciding what *should* be shown. The MVP scoping (§4) narrows by feasibility, not by importance hierarchy.

**5. Data goes stale silently.** A collector breaks (GitHub token expires, npm API changes, tasuku file format evolves) and the dashboard keeps showing the last-collected data without any indication that it's stale. James sees "npm downloads: 450/week" and doesn't realize that number is 3 months old. **Design choice that invites this:** the plan mentions no staleness detection or data-age indicators. The self-observation framing (#9) from the orchestrator was supposed to cover this, but the exploration doc's architecture (§3) has no health-check or freshness-monitoring layer for collectors.

## F. The Anti-Feature List

**1. Don't build real-time refresh.** Polling every 60 seconds (exploration doc line 387) is solving a problem that doesn't exist. None of the data sources change faster than daily (GitHub traffic is daily, npm downloads are weekly, the coverage matrix is per-release, tasuku tasks change when James works on them, at which point he's not looking at the dashboard). Real-time refresh adds WebSocket complexity and gives the illusion that the dashboard is "live" when the underlying data is not. Build it as a static report that regenerates on demand or on a schedule.

**2. Don't track social media mentions.** The exploration doc lists Reddit/HN/Twitter monitoring (line 115) and correctly flags it as hard and low-priority, but it should be explicitly killed, not deferred. Social mention monitoring is unreliable (API access is expensive or nonexistent), noisy (most mentions are low-signal), and creates anxiety (seeing a negative comment with no actionable response). Set up a GitHub Alert or Google Alert manually; don't build it into the dashboard.

**3. Don't show vanity metrics without context.** Star count, fork count, and npm downloads as standalone numbers are vanity metrics. 500 stars means nothing without knowing the category baseline. 1000 downloads/week means nothing without knowing how many are CI bots vs. real installs. If you show these, show them comparatively (vs. competitors, vs. category median) or don't show them at all. Bare numbers invite misinterpretation and emotional reactions that don't lead to good decisions.

**4. Don't build a general-purpose PM tool.** The tasuku integration panels (exploration doc §1.7, lines 73-83) are a rendering of tasuku data, not a replacement for tasuku. The dashboard should not have task-creation, task-editing, or task-transition UI. The moment you can edit tasks from the dashboard, you have two PM interfaces that can diverge, and tasuku's file-based model (which agents depend on) becomes a secondary data store.

**5. Don't optimize for mobile.** The plan mentions "mobile access via Tailscale" (exploration doc line 354). A solo-maintainer dev dashboard accessed from a phone is a recipe for anxiety-checking at dinner. If the dashboard's job is to support work decisions, it should be available where James works (desktop, terminal), not everywhere James exists. Mobile access solves no real problem and creates a bad habit.

**6. Don't expose raw API data.** Every panel should show a conclusion, not a data dump. "47 endpoints, 28 covered, 12 feasible-uncovered, 7 impossible" is a conclusion. A 47-row table with status columns is a data dump. The coverage matrix specifically (exploration doc §1.1) risks becoming a giant table that requires interpretation rather than a prioritized "here's what to build next" view. If a human has to do mental arithmetic to extract meaning, the panel has failed.

**7. Don't add competitor tracking that requires manual research.** The plan includes periodic competitor re-scans (exploration doc line 358) and ecosystem positioning (line 129). Manual competitor analysis is valuable (the awkoy comparison doc proves this), but automating it into the dashboard creates a commitment to keep it current. Competitor feature sets change unpredictably; an automated scan can check tool counts and npm stats but can't assess quality or architectural differences. Show the static comparison from the last manual review with a "last updated" date. Don't pretend it's live.
