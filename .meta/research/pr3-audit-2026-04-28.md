# PR3 audit — atomic `replace_content` + `update_block`

**Date:** 2026-04-28.
**Audit branch:** `mcp-wt-pr3-audit-2026-04-28`.
**Audit subject:** `mcp-wt-pr3-builder-2026-04-28` at `f80173a`.
**Plan:** `dev:.meta/plans/pr3-atomic-replace-update-block-2026-04-28.md` (DPs 1-6 = A/A/A/A/B/A; circuit breaker on DP1 did not fire per `pr3-live-probe-findings-2026-04-28.md`).
**Method:** four parallel Codex sessions specifying contracts to verify or break. Auditor (PM) spot-checked the most consequential findings independently before promoting them to this memo.

This memo recovers the discipline that was missing from the original builder cycle: the builder shipped 2169 lines of code+tests in a single 134-turn dispatch with no independent-review pass and no Codex mutation testing. The role-loading bug that prevented Codex pressure-testing has since been fixed; this audit is the first run with the discipline back in place.

## Headline

**Verdict: ship-with-fixes.** The `update_block` tool is sound — type-mismatch ordering, archived-flag wire shape, and per-type payload shapes are all verified correct under Codex's mutation tests, and the unit tests pin the contract well enough to catch the obvious regressions. The atomic `replace_content` rewiring is also wire-correct and consistent with `find_replace`'s warning shape. **The translator (`src/markdown-to-enhanced.ts`) is the weakest surface and the source of every High-severity finding.** Two High findings concern translator correctness (callout color/severity loss, body XML escaping); one concerns coverage (`table_row` falls to `default` and emits an `unrepresentable_block` warning that was supposed to be dead). One High finding lives in the e2e tests: the F2 rewrite dropped the failure-safety assertion (rejected `replace_content` leaves page unchanged) that the new atomic contract should now satisfy. None of the four Highs are the kind of bug that destroys data on the happy path, but each of them silently regresses a contract claim.

`update_block` and the atomic-replace handler are mergeable today with the documentation and CHANGELOG fixes (Mediums #1, #2, #5 below) plus an honest scoping of the translator's claims (a docs change, not a code change). The translator correctness fixes (Highs #1-3) and the missing F2 atomic-failure-safety test (High #4) are the pre-merge work; everything else can be deferred to follow-up tasks.

## Severity-rated findings

### Critical
None.

### High

**H1. Callout color/severity loss in translator.**
`src/markdown-to-enhanced.ts:149-154` emits `<callout icon="...">` with no `color` attribute. `pr3-live-probe-findings-2026-04-28.md:80-92` documents the required mapping (`[!TIP]` → `green_background`, `[!WARNING]` → `yellow_background`, `[!ERROR]` → `red_background`, etc.) and Notion's spec at `developers.notion.com/guides/data-apis/enhanced-markdown` accepts both `icon` and `color` on `<callout>`. Without color, every alert variant lands as a default-color callout. Severity for users: GFM alerts encode severity; the round-trip silently loses the severity dimension. **Codex evidence:** Codex grepped the spec and the probe-findings memo and produced the mapping table at the translator audit's section 2; auditor confirmed the implementation by reading lines 149-154. **Fix:** add color attribute, derive from the parsed callout's color (when `markdownToBlocks` emits one) or from the alert marker mapping (when the input was a GFM alert). **Blocking?** Yes pre-merge for Exit-A correctness — DP1=A specifically chose the translator path to preserve the convention surface. Deferring this finding regresses on that convention surface for every page round-tripped post-PR3.

**H2. XML/body escaping is incomplete in translator.**
`src/markdown-to-enhanced.ts:49-50` defines `escapeAttr` for `&` and `"` only, used inside attribute values. `richTextToEnhanced` (lines 53-67) does NOT escape any characters in body text. Inside `<callout>...</callout>`, `<details><summary>...</summary>...`, `<table><tr><td>...</td></tr></table>`, and `<column>...</column>` blocks, a user paragraph containing literal `</callout>`, `</td>`, or any `<tag>` corrupts the Enhanced Markdown XML structure. Notion's parser will misinterpret the document. The Enhanced Markdown spec (per Codex's WebFetch read) calls out `<`, `>`, `{`, `}`, `|`, `$`, `[`, `]` as needing escape outside code blocks. **Codex evidence:** Codex ran a /tmp scratch script that fed `</callout>` literal into a callout body and observed the translator emitted the literal closing tag, confirming the structural break. **Fix:** add a body-context escaper (HTML-style entity escapes inside XML containers) that runs alongside `richTextToEnhanced`. **Blocking?** Yes pre-merge if user-supplied content can contain XML-ish text — this is a correctness bug on user-content that's reasonably common (SQL blocks, JSX in prose, "</close>" comparisons in technical writing).

**H3. `table_row` is in `SUPPORTED_BLOCK_TYPES` but the translator has no case for it.**
`src/server.ts:198-204` lists `table_row` as a supported type. `src/markdown-to-enhanced.ts:195-208` handles rows ONLY as children of `case "table"`. A direct top-level `table_row` falls to `default` at line 232 and emits a `unrepresentable_block` warning. Practical impact: probably zero in current usage because `markdownToBlocks` only produces `table_row` inside a `table` parent — but the warning code itself was advertised in plan §6.3 as "dead today, defensive only." This finding turns the dead code into not-dead-code on a pathological parser path, AND reveals a coverage gap in the translator's "handles every supported type" claim. **Codex evidence:** translator audit section 1 coverage table; auditor confirmed by reading the switch. **Fix:** either add an explicit `case "table_row"` that produces a `<tr>` row outside the `<table>` wrapper (semantically dubious), or — better — narrow the translator's documented coverage to "blocks `markdownToBlocks` actually produces" and remove `table_row` from any "all supported types" claim in tests and CLAUDE.md. **Blocking?** No, but it's evidence the "translator covers everything" claim was overstated. Recommend documentation fix pre-merge plus a tasuku follow-up to widen test coverage of `file`, `audio`, `video`, `table_row`.

**H4. F2 e2e rewrite dropped the failure-safety assertion.**
The old F2 test on `dev` at `tests/e2e/live-mcp.test.ts:909` (titled "F2: replace_content with oversize payload — current destructive behavior") proved the destructive contract: an oversize-payload rejection leaves the page emptied (delete-then-append's partial-failure mode). The atomic contract should pin the OPPOSITE: rejected `replace_content` leaves existing content unchanged (atomicity = no partial state). The rewrite at builder `:907` ships only the happy-path assertion (`expect({success: true})`); no test pins the new failure-safety guarantee. The atomicity claim is in the tool description (`src/server.ts:738`) and CHANGELOG but NOT pinned by any test. **Codex evidence:** Codex diffed F2 old vs new and confirmed; auditor confirmed by reading both. **Fix:** add a live e2e that sends an invalid `replace_content` (e.g. unsupported language code block, oversize payload, rate-limit hit) and asserts the existing page content is unchanged after the rejection. **Blocking?** Yes pre-merge — atomicity is the second-most-important wedge claim after block-ID preservation; if it doesn't hold under failure, the whole migration is a regression on the existing destructive-warning contract (which at least named the partial-failure mode).

### Medium

**M1. Inline-comment preservation claimed but untested.**
`src/server.ts:738-739` and `CHANGELOG.md:14-15` claim inline-comment threads survive both `replace_content` and `update_block`. No test creates a page with an inline comment and verifies survival. Plan §7.3 item 12 specifically called this out as a Codex pressure-test target, "something the builder couldn't easily set up." **Recommended fix:** dispatch a follow-up Codex live e2e that uses a Notion sandbox to create a paragraph + inline comment, exercises `update_block` and `replace_content`, and asserts the comment thread persists by querying the comment endpoint. **Blocking?** No — claim should not be removed (probe 4 makes it plausible) but should be flagged as "untested in CI; relies on Notion server semantics" until the follow-up ships. Either soften the language in the tool description ("Notion preserves inline comments on matched blocks; not exercised in CI") or commit to running the follow-up before the next release.

**M2. `child_page` / `synced_block` non-preservation documented but untested.**
`src/server.ts:741` and `CHANGELOG.md:17-21` enumerate the types that get destroyed. No test constructs a page with one of these types and observes the actual behavior post-replace. The probe-findings memo confirms the current parser doesn't represent these types; what it does NOT confirm is what `pages.updateMarkdown`'s `replace_content` does when the source page contains them. The "they get destroyed" claim is inferred. **Fix:** another live-e2e Codex pressure test, or downgrade the claim to "Notion's replace_content drops blocks the input markdown can't represent" without enumerating types we haven't directly probed. **Blocking?** No, but the documentation is currently making an empirical claim it can't back up.

**M3. Block-ID preservation test is weaker than the live-probe claim.**
F5 at `tests/e2e/live-mcp.test.ts:1000-1003` accepts `>= 0.7` survival — would pass even if block-ID preservation regressed from 10/10 to 7/10. Probe 4 reported 10/10 on near-identical content. The relaxed bar is intentional ("resilient to small Notion-side matching changes") but doesn't constrain the wedge claim. **Fix:** keep the relaxed bar for the noisy mixed-block-types case, but add a tighter test (>= 0.95 or strict 10/10) for the homogeneous paragraph case probe 4 actually measured. Or document the relaxation in the tool description ("Notion preserves block IDs for the majority of unchanged blocks; rate may vary on complex pages"). **Blocking?** No.

**M4. `processFileUploads` runs before type-mismatch validation in `update_block`.**
`src/server.ts:1502-1510`: file uploads happen before `buildUpdateBlockPayload` rejects media block types. A user calling `update_block` with markdown that contains a `file://` token whose result is an `image`/`file`/`audio`/`video` block (none of which are in `UPDATABLE_BLOCK_TYPES`) will incur a successful Notion file upload, then receive an error. Result: a hanging Notion file upload object that's never linked to any block. **Fix:** move the type-mismatch check before `processFileUploads`, OR explicitly reject `update_block` markdown containing file tokens with a friendly error. The current plan was that `update_block` doesn't support file uploads (no tool description claim) but the handler accepts the input shape. **Blocking?** No, but it's a wasted-upload + orphan-state bug that compounds over time.

**M5. CLAUDE.md tool-registry note is missing.**
Plan §13 #4 mandates a CLAUDE.md note. `git show mcp-wt-pr3-builder-2026-04-28:CLAUDE.md | grep -i update_block` returns no hits. The architecture comment "28 tools" at `CLAUDE.md:74-75` is stale — the registry now has 29. **Fix:** add the `update_block` row to the tool registry table; bump the count. **Blocking?** Yes pre-merge per plan discipline; this is a 5-minute docs change.

**M6. Tool registration tests only count tools.**
`tests/create-page-from-file.test.ts:223-238` and `tests/http-transport.test.ts:142` were bumped from 28→29 / 27→28 — pure count assertions. No test asserts `update_block` is in the tool list by name, and no test exercises its JSON-Schema (block_id required, exactly-one-of markdown/archived). **Fix:** add a positive-inclusion test (`expect(toolNames).toContain("update_block")`) and a schema-shape test. The mutual-exclusion of markdown/archived is enforced at runtime only; a JSON-Schema `oneOf` would surface it at registration. **Blocking?** No.

**M7. Media block translator forms are wrong vs spec.**
`src/markdown-to-enhanced.ts:225-230` emits a bare URL for `file`, `audio`, `video` block types. Per the Notion Enhanced Markdown spec (Codex's WebFetch read), these need `<file src="URL">Caption</file>`, `<audio src="URL">Caption</audio>`, `<video src="URL">Caption</video>`. Practical effect: media blocks on the source page become bare-URL paragraphs after round-trip — same regression class as the bookmark/embed best-effort, but undocumented. The translator doesn't emit a warning for these (unlike bookmark/embed). **Fix:** either add the proper XML form (with caption preservation), or treat them as best-effort paragraphs and emit a warning code (`media_lost_on_atomic_replace` or similar) for parity with bookmark/embed. **Blocking?** No, but the silent regression is asymmetric with how bookmark/embed are handled.

**M8. Block-tree intermediate (the plan deviation) loses information.**
Plan §6.1 specified a text-to-text translator (`gfm: string → enhanced: string`). Builder deviated to `gfm → markdownToBlocks → serializeBlocks`. The deviation reuses the parser the server has hardened against, but the block tree drops information that a text-to-text translator would have preserved:
- `#### H4` becomes `### H4` (parser collapses everything > h2 into h3 at `markdown-to-blocks.ts:451-459`; Notion's Enhanced Markdown spec supports h4 directly).
- Multi-block callouts lose child blocks (parser's `blockquoteToBlock` filters paragraphs-only at `markdown-to-blocks.ts:188-221`; spec accepts callout children).
- Nested `+++` toggles flatten to one toggle plus trailing literal text (parser's `splitCustomSyntax:336-350` closes on first bare `+++`).
- Multiline GFM quotes serialize as one quote with embedded newlines instead of separate quote blocks plus `<br>` per the spec.
- Existing Enhanced Markdown input (e.g. `<callout icon="🔥">` from a previous round-trip) parses to zero blocks and serializes to empty.

**Codex evidence:** Codex constructed concrete inputs in /tmp and observed each loss. **Fix:** none short-term — the text-to-text path was rejected for good reason (would re-implement Notion's parser). Document the limits in the tool description and translate-warnings, file a `notion-translator-text-to-text-evaluation` follow-up to revisit if the parser-mediated path keeps producing losses. **Blocking?** No, and this is the part of the plan deviation James probably already accepted. The fix is an honest accounting in docs, not a rebuild.

**M9. CHANGELOG version-bump rationale is implicit; "duplicate_page" is overstated.**
`CHANGELOG.md` is under `[Unreleased]`, no v0.6.0 cut. The note at line 20 — "Use the Notion UI for those types or duplicate_page to preserve them" — overstates: `duplicate_page` is itself a Notion API operation that may not preserve `synced_block` references either (depending on workspace settings, etc.). **Fix:** strike the `duplicate_page` reference unless directly verified, and explicitly cut the version bump to v0.6.0 with the justification "tool API surface adds + behavioral shift on `replace_content`." **Blocking?** No.

### Low

**L1. Translator's `unrepresentable_block` warning is dead code.**
`src/markdown-to-enhanced.ts:31` defines the warning code; only emitted in the default branch (line 232). Current parser paths can't reach it (every type `markdownToBlocks` produces has a switch case, with the H3 caveat about `table_row`). **Fix:** keep as defensive (parser may grow new types) but document in the type union that this warning is rare. Or remove until needed.

**L2. README is stale on `replace_content`.**
`README.md:264` still describes `replace_content` as "destructive; duplicate_page first." Tool description and CHANGELOG describe the new atomic behavior. **Fix:** update the README. **Blocking?** No, but ship it before the v0.6.0 announcement.

**L3. `archived: false` falls through to markdown-update path.**
`src/server.ts:1465-1467` treats only `archived === true` as the archive mode. A user passing `{ block_id, archived: false }` (no markdown) hits the "provide markdown or archived: true" error correctly, but `{ block_id, markdown: "...", archived: false }` proceeds as a markdown update — silently ignoring the `archived: false`. The error at line 1473 catches both keys present, so this is the "exclusive `false` flag" case only. **Fix:** trivial — explicitly reject `archived: false` if accompanied by markdown, or reword the schema. **Blocking?** No.

**L4. `checked: false` override is not directly tested.**
Implementation at `src/server.ts:314` uses `??`, which correctly propagates explicit `false`. Tests cover `checked: true` explicit and inferred-from-`- [x]`. No test fixes the `checked: false` override case. **Fix:** add one assertion. **Blocking?** No.

**L5. Code language `tsx`, `jsx`, `mjs` are not normalized.**
`src/markdown-to-enhanced.ts:170-171` passes the parsed language string through unchanged. Notion's block-API code-language enum doesn't list `tsx`, `jsx`, `mjs`. Behavior on the live API is unconfirmed. Per learning [9d7637], TypeScript was the language that got fixed previously; unclear if the broader normalization is in place. **Fix:** centralize a language normalization helper if needed; defer until a user reports the error. **Blocking?** No.

### Informational

**I1. Code-block caption is silently dropped on `update_block`.**
`src/server.ts:318-327` builds `{ code: { rich_text, language } }` only — the SDK type accepts `caption` but the handler omits it. Per Notion's "omitted fields are unchanged" semantics for PATCH, this should NOT wipe an existing caption. So this is informational only — but the omission means there's no way to *update* a code block's caption via `update_block`. Tracked behavior, not a bug.

**I2. Heading `is_toggleable` is conditional on parser output.**
`src/server.ts:254-280` only sends `is_toggleable` when `markdownToBlocks` produced it. For plain `# Heading` markdown against an existing toggleable heading, the field is omitted. Per Notion's PATCH semantics, the existing toggleable state should survive. Untested but plausibly correct.

**I3. `unrepresentable_block` warning code is undocumented in tool description.**
Tool description at `src/server.ts:738-743` mentions `unmatched_blocks` and bookmark/embed warnings but not `unrepresentable_block` or `embed_lost_on_atomic_replace` explicitly (CHANGELOG mentions them at lines 26-28). Trivial cleanup.

**I4. Tool-list ordering is not alphabetized.**
The registry has no documented ordering convention; placement of `update_block` between `find_replace` and `read_page` is plausible but untested. Not a finding, just an observation.

## Plan-deviation justification check

The builder reported three deviations:

**D1. Translator design (block-tree-based vs plan §6.1's text-to-text).**
Justification: "reuses the parser the server has already hardened against and avoids a second markdown grammar." Verdict: **accept, with explicit accounting of M8.** The text-to-text path would have meant re-implementing Notion's GFM-with-extensions parser to map onto Enhanced Markdown — strictly worse engineering. The block-tree path's information losses (heading 4 collapse, multi-block callouts, nested toggles, Enhanced Markdown input pass-through) are inherited from `markdownToBlocks`'s existing limits, not introduced by the deviation. Action: amend the tool description and translator JSDoc to enumerate these limits honestly; file a `notion-translator-text-to-text-evaluation` follow-up if losses keep biting.

**D2. Two unrelated tests bumped (`tests/create-page-from-file.test.ts`, `tests/http-transport.test.ts`).**
Justification: "necessary tool-count assertion changes." Verdict: **accept.** Codex's diff confirmed pure `toHaveLength(N)` → `toHaveLength(N+1)` bumps. No coupling, no ordering changes, no other-tool assertions modified. Per M6, the count-only test is brittle but pre-existing — it shouldn't block PR3.

**D3. F2 e2e was rewritten.**
Justification: "the old F2 was about the destructive contract, which no longer applies." Verdict: **partial accept — see H4.** The destructive partial-failure assertion no longer applies, but the *atomic* failure-safety assertion that should replace it (rejected replace leaves page unchanged) was NOT added. The rewrite is incomplete, not just substituted.

The post-merge follow-ups (file as tasuku tasks):

- **(M1, M2)** Live e2e for inline-comment preservation and `child_page`/`synced_block` non-preservation. Group as `pr3-followup-live-coverage`.

The Highs are real but bounded: H1 and H2 are translator-correctness bugs that affect real user content; H3 is a mostly-cosmetic coverage gap; H4 is a missing test for a contract that the implementation may already satisfy. None destroy data on the happy path. Together they justify a single follow-up commit on the builder branch — or, if James prefers a clean PR, a stack of two: one for the translator + e2e fix, one for docs/CHANGELOG. ## Audit areas not covered

- **No live Notion sandbox probes during this audit.** The builder ran `scripts/bench/pr3-live-probes.ts`; this audit relied on Codex inspecting that script and the resulting findings memo rather than re-running. Specifically, M1 (inline-comment survival), M2 (`child_page`/`synced_block` destruction), and the strict block-ID preservation rate were not re-probed. The orchestrator may want to dispatch one live-probe Codex session before merge to close those.
- **Mutation tests on the translator's switch cases were proposed but not run.** Codex predicted whether each mutation would be caught by reading the test file; it did not actually mutate the source and run the tests. Predictions are reasonable but unverified.
- **The atomic-replace partial-failure path (network drop, HTTP 502 mid-stream) is unverified.** Both `replace_content` and `find_replace` rely on the outer `enhanceError` wrapper; behavior under interrupted-but-partial responses is inferred from SDK types, not tested.
- **The translator's interaction with `processFileUploads` is unaudited.** `processFileUploads` runs before the translator on `replace_content`; if a file upload produces an `image` block, the translator's image case at `:220-224` emits `![](url)` — which the live probe didn't exercise (probes used pre-existing pages with no file uploads).
- **The interaction between `update_block` and pages with mixed `synced_block` content was not probed.** Plan §7.3 item 12 was deferred to Codex; this audit did not run a live test.
