# easy-notion-mcp Roadmap

**Date:** 2026-04-23
**Status:** Working roadmap, living doc. Revised at least after each landed PR.
**Scope:** Next 2-4 weeks of wall-clock. Anything past that is intent, not commitment.

## How this doc works

Themes group work by what it is *about*. Sequence orders it by dependency and energy.

Time labels use realistic wall-clock ceilings, not honest-hours estimates. Per learning `[0186bc]`, planner plus builder plus Codex chains routinely compress 10-40x against planner time estimates (observed data from 2026-04-21: e2e-teardown 20m actual vs 4h20m planned, PR1 property-type-gap 1h15m vs 6 dev-days). "Today" means a focused working block. "This week" means 2-5 focused blocks across the week. "Next week" means 5-10 blocks the following week.

The roadmap does not replace tasuku. Tasuku tracks individual tasks with metadata; this doc is the prioritization framing over them.

### B. Agent ergonomics (the jungle gym)

Measuring how well Claude actually uses our tool surface on realistic tasks. This is the priority James called out for the roadmap conversation.

- Bench A (local-only CI jungle gym, deterministic pass/fail, 13 scenarios covering all 28 tools via realistic workflows)
  - PR-A0: skeleton plus scenario 13 Identity Smoke (cheapest, validates framework)
  - PR-A1: mature framework plus remaining 12 scenarios, including scenario 3 which regression-guards PR2's pagination fix
- Bench B (release-gate qualitative benchmark, 30-50 scenarios, per-tool friction report diffable release-over-release)
  - Strategy-agent pass to source 17-37 B-only scenarios from Notion persona research plus tool coverage matrix
  - Build: framework plus corpus, reuses Bench A infrastructure
- Bench-to-tool-description feedback loop (later, once we have 2-3 benchmark runs of real friction data)

Planning doc at `.meta/plans/e2e-artifact-a-2026-04-23.md` (Bench A PR-A0 and PR-A1). Research at `.meta/research/e2e-agent-benchmark-2026-04-23.md` (full design space). ### E. Docs hygiene

Meta-debt from the shipping pace of 2026-04-21 and 2026-04-22. Bundle into one small PR.

- `package-lock.json` version-bump-on-release rule (prevents drift between `package.json` and lock)
- `.meta/research/` lifecycle rule (if a research note has not been referenced in 90 days and did not feed into a merged plan, delete it; keeps `.meta/` lean)

### F. Scope expansion

New tools, new block types, DX improvements. Lands opportunistically rather than on a schedule.

- New tools with concrete demand or clear wins: `export_page`, `outline_page`, `diff_page`, `link_pages`, `merge_pages`, `pagestats`, `watch_database`
- New block types: `meeting_notes` (post version-bump), inline equations
- DX fold-ins: tiered tool descriptions, fold `move_page` into `update_page`, merge `add_database_entry` into `add_database_entries`, drop `share_page` (URL already returned on create)
- Notion views API tools (blocked on `notion-version-bump-2026-03-11`; user demand confirmed today via feature-request triage)
